# 02 · 技术架构设计

## 1. 技术选型（已锁死）

| 项 | 选择 | 理由 |
|---|---|---|
| 语言 | Swift 5.9+ | 唯一合理选择 |
| 窗口层 | AppKit（NSPanel） | SwiftUI 无法控制窗口层级 / 非激活行为，必须用 AppKit |
| 视图层 | SwiftUI（经 NSHostingView 挂进 NSPanel） | 三态动画和列表用 SwiftUI 写最省事，boring.notch 同款结构 |
| 持久化 | JSON 文件 + 图片文件目录 | 零依赖、可调试（能直接打开看）、500 条量级性能足够 |
| 全局快捷键 | Carbon `RegisterEventHotKey` | 唯一不需要额外权限的系统方案，Maccy 同款 |
| 最低系统 | macOS 14.0 | 带刘海机型全部可运行 macOS 14；简化 SwiftUI API 选择 |
| 第三方依赖 | 无 | 降低实现方出错面 |

## 2. 模块划分

```
│   ├ IdleNotchView   常态黑块                      │

└─────────────────────────────────────────────────┘
```

模块间依赖关系（只允许上层依赖下层，禁止反向）：

- `ClipboardMonitor` → 回调闭包产出 `ClipItem`，不知道 Store 的存在
- `AppDelegate` 负责装配：monitor 的回调里调 `HistoryStore.add()` 和 `NotchViewModel.showHUD()`
- `PanelView` 只依赖 `HistoryStore`（读）和回调（粘贴动作），粘贴动作由 Controller 层转给 `PasteService`

## 3. 数据模型

```swift
enum ClipItemType: String, Codable {
    case text      // 普通文本
    case link      // http/https URL
    case color     // #RGB / #RRGGBB / #RRGGBBAA
    case image     // 图片（内容存文件，模型里只存文件名）
    case file      // 文件引用（存路径）
}

struct ClipItem: Identifiable, Codable, Equatable {
    let id: UUID
    let type: ClipItemType
    var text: String?          // text/link/color/file 的内容（file 存路径）
    var imageFilename: String? // image 类型：images/ 目录下的文件名
    var sourceAppBundleID: String?
    var sourceAppName: String?
    let createdAt: Date
    var isPinned: Bool = false
}
```

**类型判定顺序（必须按此顺序，先命中先得）**：文件 URL → 图片 → 字符串（字符串再细分：颜色正则 → URL 解析成功且 scheme 为 http/https → 普通文本）。

## 4. 存储布局

```
~/Library/Application Support/NotchClip/
├── history.json     ClipItem 数组（含 pinned），按 createdAt 倒序
└── images/
    └── <uuid>.png   图片原始数据
```

规则：

- 写入采用「防抖保存」：内存变更后 0.5 秒内合并写盘一次（避免连续复制时频繁 IO）。
- 条数上限（默认 500）只淘汰**未固定**的最老条目；淘汰 image 条目时同步删除对应 png 文件。
- 连续两次复制内容完全相同（text 相等或图片字节相等）→ 不新增，只把旧条目提到最前（更新 createdAt）。

## 5. 关键技术决策（ADR，禁止更改）

**D1 · 剪贴板监听用轮询而不是通知。** NSPasteboard 没有变更通知 API，这是 macOS 平台限制，所有同类产品（含 Maccy）都用 Timer 轮询 `changeCount`。间隔定为 0.3 秒，CPU 开销可忽略。

**D2 · 窗口用 NSPanel + `.nonactivatingPanel`，层级高于菜单栏。** 层级取 `NSWindow.Level.mainMenu + 1`，加 `.canJoinAllSpaces` 和 `.fullScreenAuxiliary`，保证全屏应用和所有空间下都可见。这是 boring.notch 验证过的组合。

**D3 · 常态下窗口 `ignoresMouseEvents = true`。** 窗口常驻一个固定大小的容器（640×520，顶部居中盖住刘海），常态和 HUD 态下完全忽略鼠标，点击穿透到下层（否则会挡住菜单栏点击）；仅面板态接收鼠标。悬停检测不靠窗口自身，而靠全局 mouseMoved 监听器判断鼠标是否进入刘海矩形（此监听不需要任何权限）。

**D4 · 面板允许成为 key window，但粘贴前必须先归还焦点。** 搜索框要打字，面板必须能成为 key。因此：呼出面板前先记录 `NSWorkspace.shared.frontmostApplication`；粘贴时按顺序执行「关面板 → activate 原应用 → 延迟 150ms → CGEvent 发送 ⌘V」。**这是全项目第一大坑**，顺序错了粘贴就会打到自己身上。

**D5 · 自己写回剪贴板时要跳过一次监听。** PasteService 写回剪贴板会触发 changeCount 变化，若不处理，历史里会出现重复条目。用 `monitor.ignoreNextChange = true` 标志位解决。

**D6 · 不开 App Sandbox。** 模拟 ⌘V 需要辅助功能权限，直接分发 + 关沙盒是最省事且无争议的组合。Signing & Capabilities 里删除 App Sandbox 能力。

**D7 · 隐私过滤是硬规则。** 剪贴板 types 中含 `org.nspasteboard.ConcealedType` 或 `org.nspasteboard.TransientType` 的内容一律丢弃，不入库、不显示 HUD。这是密码管理器生态的通行标记。

**D8 · 图片在 UI 层按需生成缩略图。** history.json 不存图片数据，只存文件名；面板行内缩略图用 NSImage 加载后缩放显示，HUD 中同理。避免常驻内存膨胀。

## 7. 两条核心流程

**复制流程**：用户在任意应用 ⌘C → Monitor 下一个 tick 发现 changeCount 变化 → PrivacyFilter 判定（隐私标记？排除应用？）→ 解析为 ClipItem → HistoryStore.add（去重/裁剪/防抖存盘）→ NotchViewModel.showHUD(item)（1.2 秒后自动回 idle）。

**粘贴流程**：用户按 ⌘⇧V → Controller 记录 frontmostApplication → 面板展开、makeKey、聚焦搜索框 → 用户回车或 ⌘数字 → 关面板（orderOut，不 hide app）→ monitor.ignoreNextChange = true → 条目写回 NSPasteboard → 原应用 activate() → 150ms 后 CGEvent 发 ⌘V → 完成。
